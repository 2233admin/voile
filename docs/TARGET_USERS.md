# 目标用户与核心使用场景

> Refs XAR-54

---

## 用户分层

### 第一梯队：核心用户

| 用户类型 | 画像 | 最痛的记忆整理问题 | 高频输入类型 |
|----------|------|-------------------|-------------|
| 独立研究员 | 学术/行业研究者，日常在多个群组收集信息、讨论观点 | 群聊中的关键洞察散落在数百条无关消息里，手动摘录耗时且遗漏 | QQ/微信群聊、论文 PDF 链接、网页文章 |
| 独立开发者 | 参与多个技术社区，在群里讨论方案、排查问题 | 技术讨论和决策过程淹没在日常闲聊中，事后无法追溯"当时为什么选了方案 A" | QQ 技术群、GitHub issue 链接、技术博客 |

### 第二梯队：高价值用户

| 用户类型 | 画像 | 最痛的记忆整理问题 | 高频输入类型 |
|----------|------|-------------------|-------------|
| 小团队负责人 | 3-10 人团队，用 QQ/微信群做日常沟通 | 项目决策、任务分配、进度讨论混在一起，复盘时找不到关键节点 | 微信工作群、QQ 项目群、共享文档链接 |
| 内容工作者 | 自媒体/写作者，从多个信息源收集素材 | 灵感和素材来源记不住，写文章时花大量时间重新搜索已经见过的内容 | 微信公众号文章、QQ 频道帖子、网页链接、PDF |
| 咨询顾问 | 与多个客户/项目并行沟通 | 不同客户的需求、反馈、决策散落在不同群组，项目交接时信息断裂 | 微信客户群、QQ 会议记录、文档附件 |

### 非目标用户

| 类型 | 原因 |
|------|------|
| 企业 IT 管理员 | Voile 不做企业合规、审计、权限管理 |
| 纯聊天备份需求 | 已有 WeChatMsg 等工具，Voile 的价值在分析而非备份 |
| 需要云端协作的大团队 | Voile 是本地优先架构，不提供多人实时协作 |

## 高频输入类型

| 输入类型 | 来源 | 处理方式 | 优先级 |
|----------|------|----------|--------|
| QQ 群聊消息 | NapCatQQ (OneBot v11) 实时推送 | Go gateway 接收 -> Python 入库 | P0 (MVP) |
| 微信群聊消息 | WeFlow HTTP API 实时/WeChatMsg 历史导出 | Go gateway 接收 -> Python 入库 | P0 (MVP) |
| 网页链接 | 消息中提取的 URL | Go link fetcher 抓取正文 -> 入库 | P1 |
| PDF 文档 | 用户手动导入或消息附件 | Python importer 解析 -> 入库 | P2 |
| 图片中的文字 | 消息附件（截图/照片） | Rust kernel OCR（未来） | P3 |
| Telegram 消息 | Telegram Bot API | Go gateway 新 adapter（未来） | P2 |

## 核心使用场景

### 场景 1：技术讨论自动归档

| 维度 | 描述 |
|------|------|
| 触发条件 | 用户在 QQ 技术群参与了一次架构讨论 |
| 输入 | QQ 群聊消息流（10-50 条相关消息 + 穿插的无关闲聊） |
| 处理 | 话题提取 -> 情感分析 -> 参与者画像更新 -> 决策点标记 |
| 输出 | Obsidian vault 中一篇"2026-04-12 架构讨论：Redis vs Kafka"笔记，含参与者、论点、结论、原始消息链接 |
| 用户价值 | 三个月后做技术选型时，搜索"Redis Kafka"直接找到当时的讨论和结论，不用翻聊天记录 |

### 场景 2：项目决策追踪

| 维度 | 描述 |
|------|------|
| 触发条件 | 小团队在微信群讨论产品方向，做出了"先做 MVP 再融资"的决策 |
| 输入 | 微信群消息 + 群内分享的竞品分析文档链接 |
| 处理 | 决策点识别 -> 上下文提取 -> 关联文档归档 |
| 输出 | 决策日志条目 + 关联的竞品分析文档存档 |
| 用户价值 | 项目复盘时有完整的决策时间线，每个决策都能追溯到原始讨论 |

### 场景 3：研究素材自动收集

| 维度 | 描述 |
|------|------|
| 触发条件 | 研究员在多个行业群中看到相关文章和讨论 |
| 输入 | 多个群的消息流 + 群内分享的文章链接 |
| 处理 | 链接抓取正文 -> 话题标记 -> 按研究主题聚类 |
| 输出 | Obsidian vault 中按主题组织的文章存档 + 群讨论摘要 |
| 用户价值 | 写研究报告时，所有相关素材已按主题整理好，附带群友的评论和讨论 |

### 场景 4：人脉与人物画像

| 维度 | 描述 |
|------|------|
| 触发条件 | 用户需要回忆"张三是谁、上次聊了什么" |
| 输入 | 该联系人在各群组的历史消息 |
| 处理 | 人物画像 agent 汇总：活跃话题、专业领域、互动频率、关键发言 |
| 输出 | Obsidian 人物卡片：基本信息 + 话题标签 + 关键对话索引 |
| 用户价值 | 社交场合或业务对接前快速回忆对方背景，不再"这人是谁来着" |

### 场景 5：跨群主题串联

| 维度 | 描述 |
|------|------|
| 触发条件 | 同一话题在不同群组被讨论（如"AI 监管政策"在行业群、技术群、投资群都有人聊） |
| 输入 | 多个群组的消息流 |
| 处理 | 跨群话题匹配 -> 观点汇总 -> 差异标注 |
| 输出 | 主题研究笔记：综合多群视角的观点图谱 |
| 用户价值 | 一个话题的多面视角自动汇聚，不用手动在各群之间切换翻找 |

## 不同用户的路径差异

| 步骤 | 独立研究员 | 独立开发者 | 小团队负责人 | 内容工作者 | 咨询顾问 |
|------|-----------|-----------|-------------|-----------|----------|
| 接入 | QQ 学术群 + 微信行业群 | QQ 技术群 | 微信工作群 | QQ 频道 + 微信公众号群 | 微信客户群 |
| 关注分析 | 话题聚类 + 链接归档 | 决策追踪 + 技术方案记录 | 决策追踪 + 任务提取 | 素材收集 + 话题趋势 | 人物画像 + 需求追踪 |
| 主要输出 | 研究主题笔记 | 技术决策日志 | 项目复盘文档 | 素材库 | 客户档案 |
| 查询方式 | 按主题搜索 | 按项目/技术栈搜索 | 按时间线浏览 | 按关键词搜索 | 按客户名搜索 |
| MVP 够用？ | 是（入库 + 链接归档 + 基础查询） | 是 | 基本够（缺任务提取） | 基本够（缺趋势分析） | 基本够（缺画像聚合） |

## 用户获取优先级

1. **独立开发者** -- 最容易触达（技术社区），最能理解产品价值，最可能贡献代码
2. **独立研究员** -- 痛点最强（信息过载），付费意愿高
3. **小团队负责人** -- 使用频率最高（日常沟通全在群里），但部署门槛需降低
4. **内容工作者** -- 市场大但需求偏轻量，可能只需要链接归档子集
5. **咨询顾问** -- 价值高但用户量小，适合作为高端场景打磨
