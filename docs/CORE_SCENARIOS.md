## S1: 原始材料入库与自动归档

**优先级: P0 -- MVP 必做**

### 流程

```
[QQ/微信消息] -> [Go gateway 接收] -> [Rust 文本清洗] -> [Python 入库]
                                                              |
[消息中的 URL] -> [Go link fetcher 抓取正文] ---------> [Python 链接归档]
                                                              |
                                                    [Obsidian 笔记生成]
```

### 详细定义

| 维度 | 描述 |
|------|------|
| 输入 | QQ 群聊实时消息 (OneBot v11 WebSocket)、微信群聊消息 (WeFlow HTTP)、微信历史导出 (WeChatMsg CSV/JSON)、消息中的 URL |
| 处理 | 1) Go gateway 接收原始消息，写入 Redis stream; 2) Python consumer 读取，调用 Rust kernel 清洗文本; 3) 清洗后入库 (SQLite/Postgres); 4) URL 提取 + Go link fetcher 抓取网页正文; 5) 话题提取 agent 将消息流切分为话题段落; 6) Obsidian writer 生成笔记 |
| 输出 | 数据库中的结构化消息记录 + Obsidian vault 中的话题笔记和链接归档笔记 |
| 用户价值 | 零手动操作，聊天消息自动变成可检索的知识资产 |

### 验收标准

- QQ 消息从发送到入库 < 3 秒
- 微信历史 10 万条导入 < 10 分钟
- 链接抓取成功率 > 90%
- Obsidian 笔记含 frontmatter (date/tags/source/participants) + 双链
- 消息与链接的关联关系可查询

### 子场景

| 子场景 | 触发 | 输出 |
|--------|------|------|
| QQ 实时入库 | 群聊消息事件 | DB 记录 + 话题笔记 |
| 微信实时入库 | WeFlow 消息推送 | DB 记录 + 话题笔记 |
| 微信历史导入 | 用户运行 importer CLI | DB 批量记录 |
| 链接归档 | 消息中检测到 URL | 链接归档笔记 + 网页正文存档 |
| 附件保存 | 消息含图片/文件 | 文件存储 + 元数据记录（MVP 只做元数据，不做 OCR） |

## S2: 按人/事件/项目/决策回溯

**优先级: P1 -- MVP 部分覆盖（按时间+关键词查询），完整版 Phase 2**

### 流程

```
[用户查询] -> [查询接口 (gRPC / Obsidian 搜索)]
                  |
        +---------+---------+---------+
        |         |         |         |
    [按人查]  [按事件查] [按项目查] [按决策查]
        |         |         |         |
    [人物画像] [时间线]  [标签聚合] [决策日志]
        |         |         |         |
        +---------+---------+---------+
                  |
           [结果呈现: Obsidian 笔记 / gRPC 响应]
```

### 详细定义

| 维度 | 描述 |
|------|------|
| 输入 | 用户的查询意图：人名、事件关键词、项目标签、决策主题 |
| 处理 | 1) 解析查询意图（关键词 / 时间范围 / 人名 / 标签）; 2) 从 DB 检索匹配消息; 3) 按查询类型组织结果（时间线 / 人物维度 / 项目维度）; 4) 返回结构化结果 |
| 输出 | 匹配的消息列表、话题笔记、人物卡片、决策日志条目 |
| 用户价值 | "三个月前张三在技术群说的那个方案" -- 10 秒内找到 |

### MVP 与完整版对比

| 能力 | MVP (Phase 1) | 完整版 (Phase 2) |
|------|--------------|-----------------|
| 按关键词搜索 | 全文搜索 | 全文 + 语义搜索 |
| 按时间查询 | 日/周/月范围 | 自然语言时间（"上个月"） |
| 按人查询 | sender 字段精确匹配 | 人物画像 agent 聚合 |
| 按项目查询 | 标签筛选 | 项目维度聚合视图 |
| 按决策查询 | 不支持 | 决策追踪 agent |

### 验收标准

- 关键词查询返回 < 500ms
- 时间范围 + 群组组合查询可用
- gRPC 查询接口文档完整

## S3: 跨文档主题研究与串联

**优先级: P2 -- Phase 2**

### 流程

```
[多个群组的话题笔记] -> [主题匹配 agent]
                            |
                    [识别同一主题的不同讨论]
                            |
                    [生成主题研究笔记]
                            |
              [Obsidian: 主题笔记 <-> 话题笔记 双链]
```

### 详细定义

| 维度 | 描述 |
|------|------|
| 输入 | Obsidian vault 中已有的话题笔记、链接归档笔记 |
| 处理 | 1) 跨群组话题匹配（向量相似度 + 标签重叠）; 2) 观点汇总（各群对同一话题的不同视角）; 3) 差异标注（共识点 vs 分歧点）; 4) 生成主题研究笔记 |
| 输出 | 主题研究笔记：综合多群视角的观点图谱，含双链到各源话题笔记 |
| 用户价值 | 一个话题的多面视角自动汇聚，替代手动在群间切换翻找 |

### 验收标准

- 能识别 2+ 个群组中讨论同一主题的话题
- 主题研究笔记含各群观点摘要 + 差异标注
- 双链正确指向源话题笔记

### 依赖

- S1 完成（话题笔记已生成）
- Rust ANN 向量搜索可用（话题相似度计算）

## S4: 项目复盘与决策轨迹追踪

**优先级: P2 -- Phase 2**

### 流程

```
[群聊消息流] -> [决策追踪 agent]
                    |
          [识别决策点: 提议/讨论/结论]
                    |
          [构建决策时间线]
                    |
          [生成决策日志笔记]
                    |
    [Obsidian: 决策日志 <-> 话题笔记 <-> 人物卡片]
```

### 详细定义

| 维度 | 描述 |
|------|------|
| 输入 | 群聊消息流（尤其是含提议、投票、确认语义的消息） |
| 处理 | 1) 决策点识别（基于语义模式："我们决定..."、"那就这样吧"、"投票结果"等）; 2) 上下文提取（决策前的讨论、论点、参与者）; 3) 决策时间线构建; 4) 关联文档链接 |
| 输出 | 决策日志笔记：决策内容 + 讨论参与者 + 正反论点 + 最终结论 + 原始消息链接 |
| 用户价值 | 项目复盘时有完整决策链条，每个"为什么"都能追溯 |

### 验收标准

- 决策识别准确率 > 70%（中文语境下）
- 决策日志含完整上下文（不只是结论）
- 时间线视图可用

### 依赖

- S1 完成
- 话题提取 agent 成熟（决策识别建立在话题分段之上）

## S5: 基于证据的总结与建议

**优先级: P3 -- Phase 3+**

### 流程

```
[用户提问: "关于 X 我们知道什么?"]
              |
    [检索相关话题/决策/人物/链接]
              |
    [LLM 生成带引用的总结]
              |
    [输出: 总结文本 + 每句话的证据链接]
```

### 详细定义

| 维度 | 描述 |
|------|------|
| 输入 | 用户的自然语言问题 |
| 处理 | 1) 语义检索相关知识片段（话题笔记 + 链接归档 + 决策日志）; 2) LLM 生成总结，每个论点附带引用来源; 3) 输出结构化回答 |
| 输出 | 带证据链接的总结文本（每个断言可追溯到原始消息或文档） |
| 用户价值 | 不是"AI 说的"，是"张三在 3 月 15 日的讨论中提到的，原始消息在这里" |

### 验收标准

- 每个总结中的断言都有至少一条证据链接
- 证据链接指向真实存在的消息/笔记
- 无证据支撑的部分明确标注为推测

### 依赖

- S1 + S2 + S3 完成
- 向量语义搜索成熟
- LLM 调用管线稳定

## 场景优先级排序与路线图

```
Phase 1 (MVP)          Phase 2              Phase 3+
+------------------+   +------------------+  +------------------+
| S1: 入库与归档   |   | S2: 完整回溯     |  | S5: 证据总结     |
| S2: 基础查询     |   | S3: 跨群串联     |  |                  |
|                  |   | S4: 决策追踪     |  |                  |
+------------------+   +------------------+  +------------------+
     |                       |                      |
     v                       v                      v
  核心管线跑通            分析能力上线           智能层完善
  "能存能查"            "能理解能关联"         "能推理能建议"
```

### 进入 MVP 的场景

| 场景 | 进 MVP 的部分 | 理由 |
|------|--------------|------|
| S1: 入库与归档 | 全部 | 系统存在的理由，不做就没有后续 |
| S2: 基础查询 | 关键词 + 时间 + 群组查询 | 存了但查不到 = 没存 |

### 延后的场景

| 场景 | 延后到 | 理由 |
|------|--------|------|
| S2: 完整回溯（人物/项目/决策维度） | Phase 2 | 依赖画像和决策 agent |
| S3: 跨群串联 | Phase 2 | 依赖向量搜索 + 话题提取成熟 |
| S4: 决策追踪 | Phase 2 | 依赖话题提取成熟 + 决策识别模型 |
| S5: 证据总结 | Phase 3+ | 依赖全部前置场景 + LLM 管线 |

### 场景间依赖关系

```
S1 (入库归档)
 |
 +---> S2 (回溯查询)
 |      |
 |      +---> S4 (决策追踪)
 |      |
 +---> S3 (跨群串联)
 |      |
 +------+---> S5 (证据总结)
```

S1 是所有场景的基础。S5 是所有场景的终极形态。中间层（S2/S3/S4）可并行开发但各有独立依赖。
